Simulating the System - Discrete Event Simulation

Simulations are scalable and cheap and represent a trade-off between scalability and precision.

discrete_event_sim represents a queue of events, sorted by the time at which they happen.

The system state is updated at each event: in particular, what changes is the time of the simulation, which keeps increasing until the upper limit time () is reached.

The implementation of the event queue exploits an efficient data structure: a priority queue (binary heap).

The Problem

MMN problem deals with multiple servers; each of them has a queue of jobs, which are:

  • reaching the queue with a frequency λ, which is the arrival rate.
  • leaving the queue with a frequency μ, which is the completion rate.

Jobs arrive and leave in a memoryless fashion, which means that the probability of seeing a new job arriving or leaving doesn’t change in time and so is not influenced by previous jobs.

The system is supposed to be in equilibrium: the probability of moving from state  to state  is the same of moving in the opposite direction.

This condition, together with setting the value of μ to one, enables the computing of a metric W which is the average time spent in the system by a job:

         (1.1)

With n servers a load balancer is needed to divide loads between servers; this entity, in our case, can act in two different ways:

  1. through random sampling, assigning loads randomly.
  2. through supermarket queueing, taking a sample of d queues and assigning the job to the least loaded server between those.

Simulations which use the supermarket model should have a lower average time spent by the jobs in the system compared to the random model.

Implementation

Multiple Queues

In this case we are dealing with multiple queues of jobs, implemented by deque data structure which comes from collections library.

So queues are implemented with a list of deques and the current running job for each of them is saved into a list where:

  • index represents the queue index,
  • value represents the currently running job id.

Arrival and Completion

Jobs arrive and leave the queue respectively with the events of Arrival and Completion.

The first schedule for a job Arrival is done at the start of the simulation and triggers a chain of schedules; the chain is due to the fact that within the processing of a job, another Arrival event is scheduled.

The simulated delay for each Arrival event is extracted from an exponential distribution (random.expovariate), based on lambda.

To keep a coherent load while implementing MMN, the rate of new job generation is raised to nλ.

Queue Choice

A job is inserted in one of the n queues and the choice is made by using random or supermarket queueing: --random flag can be specified in order to use the first one.

Along with this choice, also another parameter can be specified, which is the percentage of queues to sample for the supermarket queueing.

This parameter can be set with --queues-sample when running the script for the simulation and is needed to compute d:

In the proposed problem, a simulation is run and the purpose is to test erasure coding in an environment of N machines and events happening on them.

Implementation

Nodes and Events

The implementation relies on a list of nodes that can be in a peer-to-peer or in a client/server relationship.

Nodes can upload or download blocks of data among them.

They have some static properties such as the space, the bandwidth and the average uptime, downtime, recovery time and lifetime.

The purpose is to keep data safe for a certain amount of time.

Each node can store  blocks up to its available space and its status changes based on events that are scheduled with an exponential distribution.

When a node goes online or completes a backup or restore it will schedule a new upload and a new download.

The upload may be due to a remote block request or a block backup; on the other hand, the download can be had from a missing block restoring or again, a block backup.

When a node fails it loses all its blocks and all its partial uploads and downloads; as soon as it recovers it tries to restore  blocks from the other nodes. If it reaches  blocks restored then the erasure coding procedure allows to restore all the missing ones.

Extension 1: Lost peers

A peer is considered lost when it can’t recover  blocks anymore; in this state, a peer will never be able to recover all its blocks and so it’s considered lost.

With --extension “base”, a lost peer stops storing its blocks and only holds other peers’ blocks.

Extension 2: Allowing peers to have multiple blocks of a peer

In the standard simulation a peer can hold just one block for a peer.

--extension “advanced” flag aim is to change this behavior, allowing peers to hold more than one block for another peer.

This extension also includes the first one, so lost peers act as described before.

This behavior leads to some modifications:

  • backed_up_blocks becomes a list of lists of Nodes, which has to identify for each block the list of nodes which are holding it.
  • remote_blocks_held values are now lists because each node can contain more than one block of another peer.

The choice of the block to back up is now a mechanism which chooses the least backed up block: the aim is to fairly back up blocks in order to have roughly the same amount of replicas for each of them.

Also, the choice of the peer which will hold the backed-up block is not trivial:

  • peers that are not already holding blocks of that peer have a higher priority than the others,
  • inside the two classes deriving from this condition the order of peers is given by the free space left.

New constraints are ensured:

  • a lost peer is not able to recover its data,
  • a peer must not hold the same block twice.

Statistics

The simulation is successful when, at the end of the simulation, all nodes haven’t lost any blocks.

In order to maintain all the blocks safe, a node has to have them locally or backed up.

We aim to grasp this information at the end of the simulation by counting, for each node, the blocks which are not held locally or remotely.

Node configurable metrics are specified in configuration files.

In order to collect more information the simulation is processed on multiple configuration files.

Thousands of simulations are made for every combination of parameters: the aim is to grasp reliable statistics on the behavior of the program in its normal state and with the extensions.
